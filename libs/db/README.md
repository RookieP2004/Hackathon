# aegis-db

The single shared database package for AEGIS AI — every SQLAlchemy model and the one consolidated Alembic migration chain implementing the complete schema from `DATABASE_SCHEMA.md`. See the root `README.md`'s "Why Two Package Ecosystems" section for why this exists as one package rather than per-service migration chains.

## Layout

```
libs/db/
├── aegis_db/
│   ├── base.py           # Declarative Base, naming convention, TimestampMixin, datetime type_annotation_map
│   ├── config.py          # DB connection settings (async + sync URLs)
│   ├── enums.py            # Python mirrors of every Postgres ENUM type
│   └── models/              # One module per DATABASE_SCHEMA.md section (organization, assets, identity, ...)
├── alembic/
│   └── versions/
│       ├── 0001_extensions_and_functions.py   # CREATE EXTENSION, trigger functions
│       ├── 0002_enums.py                       # CREATE TYPE ... AS ENUM
│       ├── 0003_core_tables.py                 # Autogenerated from aegis_db.models — all 34 non-partitioned/hypertable tables
│       ├── 0004_native_partitioning.py         # incidents/alerts/notifications — RANGE partitioning by created_at
│       ├── 0005_hypertables.py                 # TimescaleDB conversion + compression + retention + continuous aggregates
│       └── 0006_triggers_and_constraints.py    # updated_at/audit/immutability triggers, shift-overlap EXCLUDE constraint
└── seed/
    ├── seed_lookups.py      # Roles, equipment/sensor/permit/maintenance types, employers
    ├── seed_demo_plant.py    # 1 plant, 3 buildings, 10 zones, 40 equipment, 100 sensors, 5 workers/users, 2 permits
    └── run_seed.py           # Entry point — `poetry run python -m seed.run_seed`
```

## Usage From a Service

```toml
# services/<name>/pyproject.toml
[tool.poetry.dependencies]
aegis-db = { path = "../../libs/db", develop = true }
```

```python
from aegis_db.models import Equipment, Zone
```

## Local Development

```bash
poetry install
poetry run alembic upgrade head          # apply the full schema
poetry run python -m seed.run_seed        # populate demo data (idempotent)
```

## Real Bugs Found Building This — Read Before Extending the Schema

Every migration here was verified by actually running it against a live PostgreSQL + TimescaleDB instance, not just reviewed. That process surfaced four genuine, non-obvious defects — documented here so the next person adding a table doesn't reintroduce them:

1. **`__mapper_args__ = {"primary_key": [...]}` does NOT create a database primary key constraint.** It only affects the ORM's identity mapping. Every hypertable/partitioned table needs `primary_key=True` set directly on *both* columns of its composite key (see any model in `aegis_db/models/timeseries.py` for the pattern). Confirmed by autogenerate silently omitting the `PrimaryKeyConstraint` clause entirely.

2. **A bare `Mapped[datetime]` does not default to `TIMESTAMPTZ`.** Without an explicit type, SQLAlchemy produces a naive `TIMESTAMP`, violating `DATABASE_SCHEMA.md` §0.4's "always TIMESTAMPTZ" rule. Fixed globally via `type_annotation_map = {datetime: DateTime(timezone=True)}` on `Base` (`aegis_db/base.py`) — every `Mapped[datetime]` column in every model gets this for free. Don't override it with a bare `DateTime()` in a new model.

3. **`sqlalchemy.Enum(SomePythonEnum)` sends the enum's *member name*, not its `.value`, to Postgres by default.** If your Python enum values are lowercase (`IN_APP = "in_app"`) but Postgres's `CREATE TYPE` also uses lowercase values, a naive `Enum(NotificationChannel)` column will try to insert `'IN_APP'` and fail. Always pass `values_callable=lambda x: [e.value for e in x]` — see any `PgEnum(...)` call in `aegis_db/models/`.

4. **A hypertable with `add_dimension()` (space partitioning) requires every unique index, including the primary key, to include the added dimension column.** `sensor_readings`' PK is `(id, recorded_at, sensor_id)`, not `(id, recorded_at)` — the missing `sensor_id` in the PK was caught only when `add_dimension('sensor_readings', 'sensor_id', ...)` rejected the original two-column PK outright.

A fifth, non-bug lesson: `CREATE MATERIALIZED VIEW ... WITH (timescaledb.continuous)` cannot run inside a transaction — wrap it in `op.get_context().autocommit_block()` in any migration that adds a continuous aggregate, or the whole migration batch fails with "cannot run inside a transaction block."
